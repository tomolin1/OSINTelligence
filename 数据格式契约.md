# 数据格式契约

> 版本: 1.0 | 维护: 架构与数据规范组  
> 原则: 格式一旦定死，全组必须遵守，任何修改需架构组统一协调。

---

## 一、爬虫吐什么字段

每个爬虫脚本处理完一篇文章后，必须输出一个 JSON 文件，格式如下：

```json
{
  "source_url": "https://attack.mitre.org/groups/G0007/",
  "source_type": "threat_report",
  "title": "APT28 - MITRE ATT&CK",
  "raw_text": "APT28 is a threat group that has been attributed to Russia's General Staff Main Intelligence Directorate (GRU) 85th Main Special Service Center (GTsSS)...",
  "language": "en",
  "crawl_time": "2026-07-06T10:00:00Z",
  "metadata": {
    "author": "MITRE",
    "publish_date": "2026-01-15",
    "tags": ["APT28", "GRU", "Russia", "Fancy Bear"],
    "word_count": 112,
    "file_path": "/data/crawler_output/mitre_g0007.json"
  }
}
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `source_url` | string | ✅ | 原始 URL |
| `source_type` | string | ✅ | 来源类型：`threat_report` / `news_article` / `social_media` / `forum_post` / `cve_database` / `cert_alert` / `blog` |
| `title` | string | ✅ | 文档标题 |
| `raw_text` | string | ✅ | **清洗后的纯文本**（去除 HTML 标签、脚本、样式，只留文字） |
| `language` | string | ✅ | `zh` 或 `en` |
| `crawl_time` | string | ✅ | 爬取时间，ISO 8601 格式 |
| `metadata.author` | string | ❌ | 原文作者/发布者 |
| `metadata.publish_date` | string | ❌ | 原文发布时间 |
| `metadata.tags` | string[] | ❌ | 原文标签列表 |
| `metadata.word_count` | int | ❌ | 纯文本字数 |
| `metadata.file_path` | string | ❌ | 本地存储路径 |

### 爬虫组注意事项

- **不要爬全量**，先爬几个精选来源验证格式
- **`raw_text` 必须干净**：去掉 HTML 标签、CSS、JS，只留纯文本
- 所有爬虫输出统一保存在 `data/crawler_output/` 目录下
- 文件名用 `{来源}_{时间戳}.json` 格式

## 二、LLM吐什么JSON格式

LLM 组收到爬虫的 JSON 后，调用大模型抽取实体和关系。**输出必须是合法 JSON**，格式如下：

```json
{
  "document_id": "a1b2c3d4e5f67890",
  "entities": [
    {
      "id": "ENT_APT_GROUP_a1b2c3d4",
      "name": "APT28",
      "type": "APT_GROUP",
      "aliases": ["Fancy Bear", "Sofacy", "Sednit"],
      "description": "俄罗斯GRU所属APT组织",
      "source_text": "APT28 is a threat group that has been attributed to Russia's GRU",
      "confidence": 0.97,
      "mentions": [
        { "sentence": "APT28 is a threat group...", "offset_start": 0, "offset_end": 5 }
      ]
    }
  ],
  "relationships": [
    {
      "id": "REL_a1b2c3d4",
      "source_id": "ENT_APT_GROUP_a1b2c3d4",
      "target_id": "ENT_ORGANIZATION_gru",
      "type": "ATTRIBUTED_TO",
      "description": "APT28归属于俄罗斯GRU",
      "source_text": "APT28 has been attributed to Russia's GRU",
      "confidence": 0.96,
      "mitre_attack_id": "G0007"
    }
  ],
  "summary": {
    "overview": "APT28是俄罗斯GRU下属APT组织...",
    "key_findings": ["APT28归属于俄罗斯GRU", "APT28自2007年活跃"],
    "threat_level": "HIGH",
    "relevant_ttps": ["T1566.001", "T1203"]
  },
  "analysis_time": "2026-07-06T12:00:00Z"
}
```

### 实体类型枚举

LLM 必须从以下类型中选择：

| 类型 | 含义 | 示例 |
|------|------|------|
| `APT_GROUP` | APT组织 | APT28, Lazarus, Kimsuky |
| `MALWARE` | 恶意软件 | LameHug, X-Agent, WannaCry |
| `TOOL` | 攻击工具 | Metasploit, Cobalt Strike |
| `CVE` | 漏洞编号 | CVE-2025-12345 |
| `TECHNIQUE` | MITRE ATT&CK技术 | 鱼叉式钓鱼(T1566) |
| `PERSON` | 人员 | 黑客、研究员、指挥官 |
| `COUNTRY` | 国家 | Russia, North Korea, Ukraine |
| `INDUSTRY` | 行业 | 政府机构、金融、能源 |
| `CAMPAIGN` | 攻击活动 | 乌克兰网络攻击行动 |
| `ORGANIZATION` | 通用组织 | GRU, Mandiant, Kaspersky |
| `TIME` | 时间实体 | 2025年3月 |

### 关系类型枚举

| 类型 | 含义 | 示例 |
|------|------|------|
| `ATTRIBUTED_TO` | 归因于 | APT28 → GRU |
| `USES` | 使用 | APT28 → LameHug |
| `TARGETS` | 瞄准 | APT28 → Ukraine |
| `DEVELOPS` | 开发 | Lazarus → WannaCry |
| `EXPLOITS` | 利用漏洞 | APT28 → CVE-2025-12345 |
| `RELATED_TO` | 通用关联 | — |
| `AFFILIATED_WITH` | 组织间关联 | Kimsuky ↔ APT43 |
| `LEADS_TO` | 攻击链顺序 | T1566 → T1203 |
| `DISCOVERED_BY` | 被发现 | APT28 → Mandiant |
| `EMPLOYS` | 雇佣 | GRU → 某人员 |

### LLM组注意事项

- **必须 try...except 重试**：如果 LLM 返回的不是合法 JSON，重新生成
- **断点续传**：已处理的文件加标记，防止重复调用浪费费用
- **先用黄金测试集调 Prompt**：调稳了再上全量数据
- 单次调用成本太贵，注意 Token 消耗

## 三、Neo4j节点长什么样

LLM 抽取的结果存入 Neo4j，节点属性和关系类型固定如下。

### 节点标签与属性

**`(:APT_GROUP)` — APT组织**
- 属性：`id`(唯一), `name`, `aliases`, `origin_country`, `active_years`, `motivation`, `description`, `first_seen`, `last_seen`, `confidence`(0~1), `source_count`
- 索引：`name`, `origin_country`

**`(:MALWARE)` — 恶意软件**
- 属性：`id`(唯一), `name`, `aliases`, `type`(trojan/ransomware/backdoor/worm), `platform`, `capabilities`[], `description`, `first_seen`, `confidence`
- 索引：`name`

**`(:CVE)` — 漏洞**
- 属性：`id`(唯一, 用CVE编号), `name`, `description`, `cvss_score`, `severity`, `affected_software`[], `disclosure_date`, `exploitation_status`
- 索引：`name`

**`(:TECHNIQUE)` — MITRE ATT&CK技术**
- 属性：`id`(唯一, 用MITRE ID), `name`, `tactic`, `description`, `platform`[]
- 索引：`id`, `name`

**`(:COUNTRY)` — 国家**
- 属性：`id`(唯一), `name`, `code`(ISO 3166-1 alpha-2)
- 索引：`name`, `code`

**`(:INDUSTRY)` — 行业**
- 属性：`id`(唯一), `name`, `sector`
- 索引：`name`

**`(:TOOL)` — 攻击工具**
- 属性：`id`(唯一), `name`, `type`, `description`, `confidence`

**`(:PERSON)` — 人员**
- 属性：`id`(唯一), `name`, `aliases`[], `role`, `affiliation`, `confidence`

**`(:CAMPAIGN)` — 攻击活动**
- 属性：`id`(唯一), `name`, `description`, `start_date`, `end_date`, `status`, `confidence`

### 关系类型（边）

| 关系类型 | 源节点 | 目标节点 | 含义 |
|---------|--------|---------|------|
| `ATTRIBUTED_TO` | CAMPAIGN / MALWARE / TOOL | APT_GROUP | 归因于 |
| `USES` | APT_GROUP / CAMPAIGN | MALWARE / TOOL / TECHNIQUE | 使用 |
| `TARGETS` | APT_GROUP / CAMPAIGN | COUNTRY / INDUSTRY | 瞄准 |
| `DEVELOPS` | APT_GROUP | MALWARE / TOOL | 开发 |
| `EXPLOITS` | APT_GROUP / MALWARE | CVE | 利用漏洞 |
| `RELATED_TO` | ANY | ANY | 通用关联 |
| `AFFILIATED_WITH` | APT_GROUP | APT_GROUP | 组织关联 |
| `LEADS_TO` | TECHNIQUE | TECHNIQUE | 攻击链 |
| `DISCOVERED_BY` | CVE / MALWARE | ORGANIZATION / PERSON | 被发现 |
| `EMPLOYS` | APT_GROUP / ORGANIZATION | PERSON | 雇佣 |

每条关系附带属性：`id`, `confidence`(0~1), `source_text`, `mitre_attack_id`(可选)

### Neo4j操作模板

```cypher
// MERGE 创建节点（推荐）
MERGE (n:APT_GROUP {id: $id})
ON CREATE SET n += $properties
ON MATCH SET n += $updated_properties

// CREATE 创建关系
MATCH (a {id: $source_id}), (b {id: $target_id})
CREATE (a)-[r:USES {id: $rel_id, confidence: $confidence}]->(b)

// 查询子图（前端渲染用，限制2层以内）
MATCH (n)-[r]-(m)
WHERE n.id = $entity_id
RETURN n, r, m
LIMIT 100
```

## 四、一句话总结

```
爬虫输出 → 统一JSON格式 → LLM抽取 → 实体+关系JSON → 存入Neo4j → API查询 → 前端渲染
```

所有格式以 `schema.json` 为准，本文档是 schema.json 的简化说明。
